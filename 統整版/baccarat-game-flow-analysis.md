## 一、流程總覽

這是一套 **真人荷官（Live Dealer）百家樂** 的一局完整流程。核心特徵：

- **由荷官端（Dealer Tool）驅動整局節奏**：開局、掃牌、確認、下一局都從荷官端發起。
- **Server 為中繼與裁決層**：轉發開局通知、回傳結果、執行結算派彩。
- **Game UI 為玩家端呈現層**：只負責倒數、收單、顯示結果與輸贏，不參與判定。
- **含結果校驗迴圈**：開牌後有「Result is Correct?」決策節點，錯誤可重掃或手動輸入，確保結果正確才送出。

### 泳道職責

| 泳道 | 角色 | 主要職責 |
|---|---|---|
| **Dealer Tool** | 荷官端工具 | 開局、掃牌讀值、確認結果、校驗、開下一局 |
| **Server** | 伺服器 | 通知開局、回傳結果、結算派彩 |
| **Game UI** | 玩家端介面 | 倒數計時、收注、顯示結果、顯示輸贏訊息 |

## 二、逐階段流程拆解

### 階段 1｜下注階段（Bet Phase）

```
[Dealer Tool] Start
      ↓
[Dealer Tool] Start New Round（自動或手動開局）
      ↓
[Server]      Notify Game Start（通知開局）
      ↓
[Game UI]     Start bet Countdown 5s（開始 5 秒下注倒數）
      ↓
[Game UI]     Stop Bet and Submit player's wager（倒數結束、停止下注、送出玩家注單）
```

重點：
- 開局可 **自動或手動**（Auto / Manual），對應連續發牌桌與人工控場兩種營運模式。
- 下注視窗固定 **20 秒**，由 UI 端倒數。
- 倒數結束即 **鎖注並送出注單**，之後進入開牌，玩家不可再改注。

---

### 階段 2｜開牌階段（Opening Card Phase）

```
[Dealer Tool] Scan Card Revealed（掃描已翻開的牌）
      ↓
[Dealer Tool] Dealer Confirm Result（若非自動流程，荷官確認結果）
      ↓
[Dealer Tool] ◇ Result is Correct?（結果是否正確）
      ├─ Yes → [Server] Send Game Result to UI
      │              ↓
      │        [Game UI] Display Game Result for 3s（顯示結果 3 秒）
      │
      └─ No  → [Dealer Tool] Re-scan or Input Result Manually（重掃 或 手動輸入）
                     ↺ 回到校驗
```

重點：
- **掃牌讀值**：系統掃描實體牌面，自動辨識點數。
- **人工確認閘門**：`if not auto-flow` — 全自動桌可略過，人工桌需荷官確認。
- **校驗迴圈**：這是流程的關鍵防呆。辨識錯誤時 **重掃或手動輸入**，直到結果正確才由 Server 送給 UI。
- **結果展示 3 秒**：確定結果後，UI 顯示開牌結果 3 秒，讓玩家看清莊/閒/和。

---

### 階段 3｜結算（Settlement）

```
[Server]      Settlement（結算派彩）
      ↓
[Game UI]     Settlement and Display Win/Loss Message（結算並顯示輸贏訊息）

[Dealer Tool] Start New Round（自動或手動開下一局）
      ↓
[Dealer Tool] End
```

重點：
- 結算由 **Server** 執行（依注單與結果計算輸贏），UI 端只顯示輸贏訊息。
- 荷官端隨即 **開下一局**（回到階段 1），形成連續循環；`End` 代表牌局結束或換靴。

## 三、時序與關鍵節點

| 節點 | 觸發方 | 時間 | 說明 |
|---|---|---|---|
| 開局 | Dealer Tool | — | 自動或手動 |
| 下注倒數 | Game UI | **20s** | 固定下注視窗 |
| 停止下注 | Game UI | 倒數結束 | 鎖注、送注單 |
| 掃牌 / 確認 | Dealer Tool | — | 含人工確認閘門 |
| 結果校驗 | Dealer Tool | — | 錯誤重掃迴圈 |
| 結果展示 | Game UI | **3s** | 顯示莊/閒/和 |
| 結算 | Server → UI | — | 派彩 + 輸贏訊息 |
| 下一局 | Dealer Tool | — | 循環或 End |

## 四、觀察與潛在缺口

以下為以百家樂 UX / 系統流程角度提出的分析，**非原圖既有內容**，供優化參考：

### 4.1 流程未涵蓋的環節

| 缺口 | 風險 | 建議補上 |
|---|---|---|
| **補牌邏輯（第三張牌）未顯示** | 百家樂莊閒可能各補一張，流程只有「Scan Card」單步 | 開牌階段細分「發前兩張 → 判定例牌 → 依規則補第三張」 |
| **下注超時行為未定義** | 玩家 20 秒沒下注會怎樣？ | 明確標注「不下注」或「沿用上局注」 |
| **斷線 / 重連未處理** | 玩家開牌時斷線，結算能否正確反映？ | 加入重連恢復狀態的分支 |
| **校驗迴圈無上限** | Re-scan 無限迴圈可能卡局 | 設定重試上限，超過轉人工主管介入 |
| **和局（Tie）呈現未細分** | 「Win/Loss Message」二元，未含和局退注 | 輸贏訊息應含「和局：莊閒退注」狀態 |
| **邊注/對子結算未提** | 若支援莊閒對、龍寶等邊注，結算需分項 | 結算階段標注各下注項獨立派彩 |

### 4.2 時序合理性

- **20 秒下注**：符合標準百家樂節奏；若做 Speed Baccarat 需縮短並精簡 UI。
- **結果展示 3 秒 → 結算**：3 秒偏短，若含派彩動畫（籌碼飛入、數字滾動）可能被截斷，建議展示與派彩動畫時間可重疊或允許跳過。
- **結算與開下一局並行**：圖中結算（Server/UI）與開下一局（Dealer）看似並行，需確認「結算完成」是否為「開下一局」的前置條件，避免玩家還沒看到輸贏就進下一局。

### 4.3 校驗迴圈是亮點

「Result is Correct?」+ 重掃/手動輸入，是這套流程最扎實的設計——確保 **結果正確性優先於速度**，符合真人荷官博弈的合規要求。建議保留並補上重試上限與稽核紀錄（誰在何時手動改了結果）。

## 六、建議的優化後流程（摘要）

```
下注階段：開局 → 通知 → 20s 倒數 →（超時＝不下注/沿用上局）→ 鎖注送單
開牌階段：發前兩張 → 判例牌 → 依規則補第三張 → 掃牌辨識
          → 確認閘門 → ◇正確? ─No→ 重掃/手動（≤N 次，超過轉主管）
                              └Yes→ 送結果 → 展示（含和局狀態）
結算階段：主注 + 邊注分項派彩 → 顯示輸贏/和局訊息 →（結算完成後）開下一局
異常處理：斷線重連恢復狀態、稽核紀錄手動改值
```
